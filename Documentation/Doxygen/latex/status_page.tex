Statuses are long‑lasting effects applied to units during gameplay. They can heal over time, deal damage over time, apply buffs, combine into new effects, or trigger logic at the start or end of each turn.

This chapter explains:\+ \begin{DoxyVerb}- What a StatusData is
- What a StatusInstance is
- How statuses are applied and removed
- How turn‑based activation works
- How status mixing works
- How to create custom statuses
- How to use built‑in statuses
- How to integrate statuses into your turn system
- How to connect statuses to UI, VFX, and audio
- Best practices
\end{DoxyVerb}
\hypertarget{status_page_autotoc_md67}{}\doxysubsubsection{\texorpdfstring{1. Understanding Statuses}{1. Understanding Statuses}}\label{status_page_autotoc_md67}
Statuses in this system are split into two layers:\+ Status\+Data (Permanent Data)

A Scriptable\+Object that defines:\+ \begin{DoxyVerb}- id
- description
- duration
- components (for mixing)
- CreateInstance()
\end{DoxyVerb}


This is the design‑time representation of a status. Status\+Instance (Runtime State)

A lightweight class that stores:\+ \begin{DoxyVerb}- A reference to the StatusData
- The current duration
- Activation hooks:

    - OnApply
    - OnRemove
    - OnTurnStart
    - OnTurnEnd
\end{DoxyVerb}


Each unit holds a list of Status\+Instances through its Status\+Manager. Key Concepts \begin{DoxyVerb}- Statuses are turn‑based
- Statuses are single‑target
- Statuses are backend‑only
- Statuses can mix into new statuses
- Statuses can be composite (multiple sub‑effects)
- Statuses can apply buffs
\end{DoxyVerb}


Statuses do not handle UI, VFX, or audio. Developers should use Ability\+System\+API events to connect statuses to presentation. \hypertarget{status_page_autotoc_md68}{}\doxysubsubsection{\texorpdfstring{2. Status\+Data}{2. Status\+Data}}\label{status_page_autotoc_md68}

 How Mixing Works

When applying a new status:\+ \begin{DoxyVerb}- Check if it mixes with any active status
- If so, remove the active status
- Replace both with the mixed result
- Repeat until no more mixes occur
- Apply the final result
\end{DoxyVerb}


Component Lists

Mix\+Rules automatically generate flattened component lists so the system knows:\+ \begin{DoxyVerb}- Which statuses are “ingredients”
- Which statuses should not be re‑applied
- How to prevent circular mixes
\end{DoxyVerb}

\hypertarget{status_page_autotoc_md75}{}\doxysubsubsection{\texorpdfstring{9. Best Practices}{9. Best Practices}}\label{status_page_autotoc_md75}
\begin{DoxyVerb}- Keep statuses small and focused
- Use mixing for emergent gameplay
- Use Composite for multi‑effect statuses
- Avoid putting UI/VFX logic inside statuses
- Use events for presentation
- Keep StatusData immutable
- Use StatusInstance for runtime state only
- Use components to prevent re‑applying ingredient statuses
\end{DoxyVerb}
