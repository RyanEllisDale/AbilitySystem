Effects are the fundamental action units of the Tactics Ability System Package. Each Effect represents one atomic gameplay action performed when an ability activates — such as healing, applying a status, pushing a unit, or raising an event.

\end{DoxyVerb}

Key Concepts \begin{DoxyVerb}- parent — the unit activating the ability

\hypertarget{effect_page_autotoc_md58}{}\doxysubsubsection{\texorpdfstring{6. Chaining Effects}{6. Chaining Effects}}\label{effect_page_autotoc_md58}
Effects are naturally chainable because:\+ \begin{DoxyVerb}- Abilities execute Effects in order
- Effects do not depend on each other
- Effects can modify the game state for later Effects
\end{DoxyVerb}


Example chain:\+ \begin{DoxyVerb}- DamageEffect
- ApplyStatusEffect
- PushEffect
- CallEvent
\end{DoxyVerb}


This allows for complex ability behaviors without writing monolithic code. \hypertarget{effect_page_autotoc_md59}{}\doxysubsubsection{\texorpdfstring{7. Integrating Effects With UI, VFX, and Audio}{7. Integrating Effects With UI, VFX, and Audio}}\label{effect_page_autotoc_md59}
Effects do not trigger visuals, animations, or UI.

This is intentional.

Developers should use Ability\+System\+API events:\+ \begin{DoxyVerb}- AbilityExecuted
- StatusApplied
- BuffApplied
- StatusActivated
- BuffRemoved
\end{DoxyVerb}


These events allow you to:\+ \begin{DoxyVerb}- Play animations
- Spawn VFX
- Play audio
- Update UI
- Trigger camera shakes
- Drive AI reactions
\end{DoxyVerb}


This keeps the Effect system clean, backend‑only, and presentation‑agnostic. \hypertarget{effect_page_autotoc_md60}{}\doxysubsubsection{\texorpdfstring{8. Best Practices}{8. Best Practices}}\label{effect_page_autotoc_md60}
\begin{DoxyVerb}- Keep Effects small and focused
- Use multiple Effects instead of one large one
- Use DataReferences for dynamic values
- Use API events for presentation logic
- Avoid hard‑coding visuals or UI inside Effects
- Use chaining to build complex behaviors
- Keep Effects stateless whenever possible
\end{DoxyVerb}
